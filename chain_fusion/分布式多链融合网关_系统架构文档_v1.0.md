## 1. 架构概述

### 1.1 背景与定位

分布式多链融合网关（Chain Fusion）是面向金砖国家银行（NDB）等多边金融机构，服务于跨境支付清算、数字资产流转、RWA 代币化等核心业务场景的**链上链下融合枢纽**。

当前金融系统面临以下核心痛点：


| 痛点         | 描述                                       |
| ---------- | ---------------------------------------- |
| 多链协议不兼容    | 公链（ETH/Conflux/BSC）与联盟链（CIPS）协议异构，无法统一接入 |
| 跨境清算效率低    | 传统 SWIFT/CIPS 链路延迟高、成本高，难以满足数字资产实时清算需求   |
| 数据安全与一致性风险 | 链上链下数据同步缺乏可信机制，存在篡改风险                    |
| 合规管控薄弱     | 多司法辖区 AML/CFT、旅行规则、制裁名单筛查能力缺失            |


Chain Fusion 作为核心基础设施，打通区块链生态与传统金融系统（CIPS、SWIFT、银行核心系统），实现数字资产交易、清算、托管全流程的高效、安全、合规流转。

### 1.2 核心设计原则


| 原则         | 说明                                              |
| ---------- | ----------------------------------------------- |
| **兼容性优先**  | 适配银行现有 IT 架构（COBOL 大型机、SAP、Temenos），银行系统无需大规模改造 |
| **多链无感知**  | 银行核心系统只对接一套标准化接口，网关屏蔽所有多链协议差异                   |
| **合规内嵌**   | 将 AML/CFT 筛查、身份映射、交易追溯等合规能力内嵌于网关层，避免架构冗余        |
| **插件化可扩展** | 新增链/协议仅需部署适配插件，维护成本线性增长，新链接入周期 ≤ 1-2 周          |
| **金融级安全**  | 满足网络安全等保三级要求，保障私钥、交易数据、身份信息的传输与存储安全             |

### 2.2 各层职责说明


| 层级          | 核心职责                             | 关键组件                                           |
| ----------- | -------------------------------- | ---------------------------------------------- |
| **银行核心系统层** | 提供传统金融业务能力，通过标准化接口与网关对接，无需改造核心逻辑 | COBOL 大型机、分布式核心系统、Open Banking                 |
| **接入层**     | 接收并校验外部请求，负载均衡，完成第一层协议格式转换       | Netty + Spring Cloud Gateway / Apache APISIX   |
| **核心业务层**   | 执行协议适配、路由决策、合规校验、安全防护、交易处理全部核心逻辑 | 六大核心子模块（详见第 3 节）                               |
| **数据层**     | 负责交易数据、合规数据、配置数据、日志数据的存储与管理      | MySQL / Redis / Elasticsearch / InfluxDB / HSM |
| **多链生态层**   | 对接各类区块链节点，提供链上资产交互与智能合约调用能力      | 公链节点、联盟链节点、跨链桥                                 |
| **支撑体系**    | 横向支撑所有层级的合规校验、运维监控、故障告警能力        | 合规中台、Prometheus、Grafana、ELK                    |

## 3. 核心模块架构

### 3.1 核心模块矩阵


| 模块层级      | 核心模块      | 核心功能                                                    | 技术选型                                                 |
| --------- | --------- | ------------------------------------------------------- | ---------------------------------------------------- |
| **接入层**   | 银行侧接入模块   | 适配 HTTP/REST、SOAP、COBOL 自定义协议，数据格式转换，请求校验与负载均衡          | Netty + Spring Cloud Gateway；自定义 COBOL 协议解析器         |
| **协议适配层** | 多链协议抽象模块  | 封装公链 RPC、联盟链接口、智能合约 ABI，抽象为标准化 API                      | Protobuf 统一数据模型；插件化适配器（JSON-RPC / Web3.js / 联盟链 SDK） |
|           | 数据格式转换模块  | 链下结构化数据（COBOL/SQL）与链上数据（JSON/ABI）双向转换，EBCDIC/UTF-8 编码互转 | 自定义数据映射引擎；支持 XML/JSON/CSV/COBOL 格式互转                 |
| **动态路由层** | 多链状态感知模块  | 实时监控链节点健康度、链上拥堵、跨链桥可用性，生成健康评分                           | 定时探测 + 事件驱动；Redis 存储链状态                              |
|           | 智能路由调度模块  | 基于交易类型、成本、效率、合规要求动态选择最优路径                               | 加权轮询算法；支持自定义路由策略                                     |
|           | 容灾备份模块    | 主节点故障时自动切换备用节点/跨链桥，保障交易连续性                              | 主备切换机制 + RocketMQ 交易重试队列                             |
| **合规控制层** | 身份映射模块    | 钱包地址与银行账户/法定身份绑定，建立四级身份映射                               | DID；对接 eIDAS、VerifySeal 等合规数字身份服务商                   |
|           | 实时合规校验模块  | AML/CFT 筛查、制裁名单比对、旅行规则适配、多辖区合规校验                        | Drools 规则引擎 + 机器学习模型；对接 OFAC/UNSC 制裁名单库              |
|           | 合规日志与追溯模块 | 记录链上链下全流程交易数据，生成可审计合规报告                                 | ELK 分布式日志 + 链上哈希存证 + 链下加密存储                          |
| **安全防护层** | 传输安全模块    | 保障网关与银行系统、多链节点的通信安全                                     | TLS 1.3；国密 SM4 / 国际 AES-256 双加密                      |
|           | 签名验证模块    | 交易签名验证，防止未授权操作                                          | ECDSA secp256k1 / 国密 SM2 双签名算法                       |
|           | 防攻击模块     | 抵御重放攻击、DDoS、SQL 注入等恶意请求                                 | 时间戳 + nonce 防重放；Redis+Lua 限流；WAF；IDS 入侵检测            |
| **运维监控层** | 监控告警模块    | 实时监控运行状态、交易成功率、响应时间、链节点状态                               | Prometheus + Grafana；AlertManager 多渠道告警              |
|           | 配置管理模块    | 支持适配插件、路由策略、合规规则的动态配置与灰度发布                              | Apollo 配置中心                                          |

## 4. 子模块详细架构

### 4.1 协议适配子模块

采用"**插件化 + 抽象封装**"三层架构：

```
外部请求（银行系统 / 区块链节点）
        │
┌───────▼─────────────────────────────────┐
│              协议抽象层                  │
│    定义统一协议接口，屏蔽异构差异          │
│    提供标准化 GatewayMessage 数据结构     │
└───────┬─────────────────────────────────┘
        │
┌───────▼─────────────────────────────────┐
│              插件适配层                  │
│  Conflux（JSON-RPC）  ETH（Web3.js）     │
│  BSC（Web3.js）       CIPS（自定义 SDK）  │
│  支持 SPI 热插拔，新增链 ≤ 1-2 周         │
└───────┬─────────────────────────────────┘
        │
┌───────▼─────────────────────────────────┐
│              协议转换层                  │
│  格式转换：JSON/ABI ↔ COBOL/XML/Protobuf │
│  编码转换：EBCDIC ↔ UTF-8                │
└─────────────────────────────────────────┘
```

**统一内部消息格式（Protobuf）：**

```protobuf
message GatewayMessage {
    string request_id     = 1;   // 请求唯一 ID
    int64  timestamp      = 2;   // UTC 毫秒时间戳
    string source_chain   = 3;   // 源链标识
    string target_chain   = 4;   // 目标链标识
    string asset_type     = 5;   // 资产类型
    string payload        = 6;   // 业务数据（JSON 序列化）
    string signature      = 7;   // 请求签名
    string bank_account   = 8;   // 银行账户（可选）
}
```

### 4.2 动态路由子模块

**健康度评分算法：**

```
健康度 = (连通率 × 40%
        + 确认延迟达标率 × 30%
        + 跨链桥成功率 × 20%
        + 合规状态 × 10%) × 100
```


| 参数     | 说明                                    |
| ------ | ------------------------------------- |
| 数据更新频率 | 普通链：10 秒/次；高优先级链（CIPS）：1 秒/次          |
| 路由优先策略 | 合规优先 → 效率优先（核心交易）/ 成本优先（非核心交易）→ 健康度优先 |
| 容灾切换时间 | 主节点故障自动切换，恢复时间 ≤ 5 分钟                 |


**路由决策流程：**

```
请求接入
    │
    ▼
请求解析（提取源链、目标链、资产类型等关键信息）
    │
    ▼
规则匹配（基于预设规则：资产类型 / 网络性能 / 合规要求）
    │
    ▼
负载均衡（在多个可用目标链节点间分配流量）
    │
    ▼
目标链选择（综合匹配结果与负载状态，智能决策最优节点）
    │
    ▼
交易转发 → 系统对接子模块
```

### 4.3 合规控制子模块

**身份映射四级体系：**

```
银行客户完成 KYC
    │
    ▼
银行账户  →  法定身份  →  DID  →  合规钱包地址
    │
    ▼（映射关系存储）
链上哈希存证（不可篡改）
    +
链下 AES-256 加密存储（仅授权节点可查询）
```

**多辖区合规规则库：**


| 辖区   | 适配规则                          |
| ---- | ----------------------------- |
| 欧盟   | GDPR、MiCA 法案、MiFID II         |
| 中国   | 《数据出境安全评估办法》、AML、eCNY 相关规范    |
| 香港   | SFC 报表要求、《个人数据（隐私）条例》、HKMA 监管 |
| 全球   | FATF 旅行规则、OFAC/UNSC 制裁名单、CFT  |
| 金砖国家 | mBridge 合规要求、NDB 相关金融监管       |


### 4.4 安全防护子模块

**安全纵深防御体系：**

```
┌─────────────────────────────────────────────────────┐
│  边界防护：WAF 拦截 SQL 注入 / XSS / DDoS             │
├─────────────────────────────────────────────────────┤
│  传输安全：TLS 1.3 + 国密 SM4（敏感字段双加密）        │
├─────────────────────────────────────────────────────┤
│  身份认证：API Key / 数字证书 / OAuth 2.0 + JWT       │
├─────────────────────────────────────────────────────┤
│  权限控制：RBAC 最小权限 + 敏感操作多签审批            │
├─────────────────────────────────────────────────────┤
│  防重放：时间戳 + nonce，请求有效期 ≤ 30 秒            │
├─────────────────────────────────────────────────────┤
│  存储安全：HSM 私钥 + AES-256 数据加密                │
├─────────────────────────────────────────────────────┤
│  签名验证：ECDSA secp256k1 / 国密 SM2 双算法支持      │
└─────────────────────────────────────────────────────┘
```

### 4.5 交易处理子模块

**跨境交易清算核心流程（9步）：**

```
① 银行内部系统发起跨境交易指令
②  接入层：请求校验 + 负载均衡 + 协议格式转换
③  协议适配子模块：协议格式转换 + 合法性校验
④  交易处理子模块：参数校验 + 权限校验 + 余额校验
⑤  合规校验子模块：制裁名单筛查 + 异常交易识别
⑥  动态路由子模块：选择最优路由路径（CIPS / mBridge / 公链）
⑦  系统对接子模块：转发交易指令至对应系统
⑧  接收交易执行结果 → 更新交易状态 → 写入数据层（链上哈希存证）
⑨  将交易结果通过接入层反馈至银行内部系统
```

**异常处理机制：**

- 基于 RocketMQ 重试队列，失败交易自动重试（最大重试次数可配置）
- 支持手动干预与回滚操作
- 完整异常日志（含异常原因、堆栈、上下文）

## 5. 数据架构

### 5.1 核心数据实体关系

```
┌──────────┐     1:N    ┌──────────┐
│  Chain   ├───────────▶│  Asset   │
│（区块链） │            │（数字资产）│
└────┬─────┘            └────┬─────┘
     │                       │
     │ N:1  source/target     │ N:1
     │                       │
┌────▼───────────────────────▼─────┐
│           Transaction            │
│         （跨链交易记录）          │
└──────────────────────────────────┘

┌──────────┐     N:M    ┌──────────┐    N:M    ┌────────────┐
│   User   ├───────────▶│   Role   ├──────────▶│ Permission │
│  （用户） │            │  （角色） │           │  （权限项）  │
└──────────┘            └──────────┘           └────────────┘
```

**核心关系：**

- `Chain 1:N Asset` — 一条链包含多种资产
- `User N:M Role` — 支持单用户多角色
- `Role N:M Permission` — 角色包含多种权限
- `Transaction N:1 Chain（source/target）` — 交易关联源链和目标链
- `Transaction N:1 Asset` — 交易关联具体资产

### 5.2 数据分类与存储策略


| 数据类型     | 存储方案                   | 留存要求  | 说明                                   |
| -------- | ---------------------- | ----- | ------------------------------------ |
| **交易数据** | MySQL 主从 + Redis（热点缓存） | ≥ 7 年 | 主库写入，从库读取；热点数据缓存提升查询性能               |
| **合规数据** | MySQL + Redis          | ≥ 7 年 | 制裁名单存 Redis 快速查询；规则和报表存 MySQL        |
| **配置数据** | MySQL + Redis          | 长期    | 写入 MySQL，缓存 Redis 快速加载；Apollo 管理动态配置 |
| **日志数据** | Elasticsearch（ELK）     | ≥ 7 年 | 支持全文检索与可视化分析                         |
| **监控数据** | InfluxDB               | 自定义   | 时序数据库，Grafana 可视化                    |
| **私钥**   | HSM 硬件安全模块             | 长期    | 严禁明文存储；密钥轮换周期 ≤ 90 天                 |
| **敏感字段** | 加密数据库（AES-256/SM4）     | ≥ 7 年 | 账户信息、身份信息脱敏加密存储                      |


### 5.3 数据流转闭环

```
外部系统（银行/区块链节点/CIPS/mBridge）
        │ 接入
        ▼
    接入层校验
        │
        ▼
  核心业务层处理
  （协议转换 → 合规校验 → 路由决策 → 交易执行）
        │
        ▼
    数据层存储
    （MySQL / Redis / ELK）
        │
        ▼（核心数据链上哈希存证）
    链上存证 + 链下加密备份
        │
        ▼ 结果反馈
外部系统（状态同步/对账/审计）
```

## 6. 安全架构

### 6.1 金融级安全体系

```
┌─────────────────────────────────────────────────────────────────┐
│                       安全管控全景                               │
├─────────────┬───────────────────────┬───────────────────────────┤
│  身份与访问  │    数据安全            │    抗攻击与合规           │
├─────────────┼───────────────────────┼───────────────────────────┤
│ KYC + DID   │ TLS 1.3 传输加密       │ WAF 边界防护               │

└─────────────┴───────────────────────┴───────────────────────────┘
```

### 6.2 加密算法选型


| 场景       | 算法              | 说明            |
| -------- | --------------- | ------------- |
| 传输加密     | TLS 1.3         | 所有通信链路        |
| 敏感字段加密   | 国密 SM4          | 身份信息、交易金额     |
| 存储加密     | AES-256 / SM4   | 交易日志、合规数据     |
| 数字签名（国际） | ECDSA secp256k1 | 标准区块链签名       |
| 数字签名（国密） | SM2             | 国内合规场景        |
| 私钥存储     | HSM 硬件安全模块      | 密钥轮换周期 ≤ 90 天 |


### 6.3 合规管控边界

- **制裁名单筛查：** 对接 OFAC、UNSC 全球制裁名单数据库，实时更新
- **旅行规则：** 交易发起时自动提取双方法定身份信息，同步至监管平台
- **匿名交易拦截：** 拒绝未绑定合规身份的匿名钱包参与交易
- **多辖区适配：** 根据交易参与方所属地区、资产类型自动匹配对应规则

## 9. 系统交互与数据流

### 9.1 跨境交易清算交互时序

```
银行内部系统      接入层        核心业务层（多子模块）        数据层      外部系统（CIPS/链）
     │              │                  │                       │              │
     │──发起交易指令→│                  │                       │              │
     │              │──请求校验+协议转换→│                       │              │
     │              │                  │──协议解析+合法性校验───│              │
     │              │                  │──合规筛查（AML/制裁）──│              │
     │              │                  │──选择最优路由路径───────│              │
     │              │                  │──转发交易指令──────────────────────→ │
     │              │                  │←─────────────────────────执行结果── │
     │              │                  │──更新交易状态────────→ │              │
     │              │                  │──链上哈希存证──────────────────────→ │
     │ ←─交易结果───│ ←─封装响应────── │                       │              │
     │              │                  │                       │              │
```

### 9.2 系统集成数据同步

```
银行核心系统（账户/余额/单据）
    │ 定时推送 / 事件触发
    ▼
系统对接子模块（接口适配层）
    │
    ▼
数据安全子模块（加密 + 脱敏）
    │
    ▼
数据层存储（MySQL + Redis + 链上存证）
    │
    ▼（实时同步至相关业务子模块）
交易处理 / 合规校验 → 结果反馈至银行
```

### 9.3 API 接口规范


| 项目    | 规范                                 |
| ----- | ---------------------------------- |
| 通信协议  | HTTPS 1.3（银行侧 → 网关）/ TCP（网关 → 链节点） |
| 数据格式  | JSON（默认）/ XML（可选）/ Protobuf（链上数据）  |
| 基础路径  | `/api/v1/gateway`                  |
| 签名算法  | ECDSA secp256k1（国际）/ 国密 SM2（国内）    |
| 时间戳   | UTC 毫秒级，请求有效期 ≤ 30 秒（防重放）          |
| 认证方式  | API Key / 数字证书 / OAuth 2.0 + JWT   |
| 错误码规范 | 按"业务类型 + 错误类型"编码                   |


**核心接口列表：**


| 接口     | 方法   | 路径                                    | 说明             |
| ------ | ---- | ------------------------------------- | -------------- |
| 资产查询   | POST | `/api/v1/gateway/asset/query`         | 查询账户在目标链上的资产余额 |
| 跨链资产转移 | POST | `/api/v1/gateway/transfer`            | 发起跨链资产转移请求     |
| 查询交易状态 | GET  | `/api/v1/gateway/transaction/{tx_id}` | 查询跨链交易当前执行状态   |
| 合规预校验  | POST | `/api/v1/gateway/compliance/check`    | 对交易发起合规前置校验    |
