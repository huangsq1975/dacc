## 1. 軟件概述

### 1.1 軟件名稱

**分佈式多鏈融合網關（Chain Fusion）**

### 1.2 軟件定位

分佈式多鏈融合網關（Chain Fusion）是面向金磚國家銀行（NDB）等多邊金融機構的鏈上鏈下融合核心基礎設施，服務於跨境支付清算、數字資產流轉、RWA 代幣化等核心業務場景。

本軟件作為連接不同區塊鏈與傳統金融系統的樞紐，打通區塊鏈生態與 CIPS、SWIFT、銀行核心系統之間的協同壁壘，實現數字資產交易、清算、托管全流程的高效、安全、合規流轉。

### 1.3 解決的核心問題

| 痛點 | 解決方案 |
|------|----------|
| 多鏈協議不兼容，無法統一接入公鏈與聯盟鏈 | 插件化協議適配，統一抽象封裝各鏈接口 |
| 跨境清算效率低、延遲高、成本高 | 智能路由調度，端到端清算延遲 ≤ 3.5 秒，成本降低 60% 以上 |
| 鏈上鏈下數據同步缺乏可信機制 | 哈希存證 + 多副本同步，確保數據一致性與防篡改 |
| 合規管控薄弱，難以滿足多司法轄區監管要求 | 內嵌 AML/CFT、制裁名單篩查，滿足多司法轄區合規要求 |

### 1.4 適用範圍

- **接入鏈類型：** Conflux、ETH、BSC 等主流公鏈，CIPS 相關聯盟鏈，合規 DeFi 平台
- **業務場景：** 跨境支付清算、RWA 代幣化發行與托管、貿易融資單據存證、跨鏈資產映射
- **目標用戶：** 金磚國家銀行等多邊金融機構、跨境金融企業、數字資產服務機構
- **對接系統：** COBOL 大型機、分佈式核心系統、Open Banking 平台、CIPS、mBridge

## 4. 功能說明

### 4.1 多鏈協議適配

本模塊採用「插件化 + 抽象封裝」三層架構，對不同區塊鏈協議進行統一封裝，屏蔽協議差異。

**主要功能：**
- **多鏈協議封裝：** 支持 Conflux（JSON-RPC）、ETH（Web3.js）、BSC、CIPS 聯盟鏈（自定義 SDK）等協議統一抽象，提供標準化接口
- **數據格式轉換：** 支持 JSON/ABI 與 COBOL、XML、Protobuf 等格式雙向轉換，支持 EBCDIC/UTF-8 編碼互轉
- **插件化擴展：** 通過 SPI 機制加載適配插件，支持熱插拔，新增鏈適配周期 ≤ 1-2 週
- **傳統金融協議適配：** SWIFT / CIPS ↔ 區塊鏈 RPC 協議適配，支持 ISO 20022 報文標準

### 4.2 動態路由與智能調度

**健康度評分算法：**

```
健康度 = (連通率 × 40% + 確認延遲達標率 × 30% + 跨鏈橋成功率 × 20% + 合規狀態 × 10%) × 100
```

**主要功能：**
- **節點狀態感知：** 實時監控連通率、確認延遲、Gas 費、跨鏈橋可用性；普通鏈 10 秒/次更新，高優先級鏈（CIPS）1 秒/次更新
- **智能路由選擇：** 路由優先策略：合規優先 → 效率優先（核心交易）/ 成本優先（非核心交易）→ 健康度優先
- **自動容災切換：** 主節點故障時自動切換至備用節點，保障業務連續性
- **動態配置：** 路由規則支持動態下發與熱更新，無需重啟服務

### 4.3 高性能交易處理

**交易處理流程：**

```
接收交易指令 → 參數解析 → 協議校驗 → 合規前置篩查 → 路由選擇 → 轉發執行 → 狀態同步 → 結果回調
```

**主要功能：**
- **高並發處理：** 支持水平擴展，常規 ≥ 1000 TPS，峰值擴展至 5000 TPS
- **交易重試隊列：** 基於 RocketMQ 實現失敗交易自動重試，確保交易不丟失
- **批量交易處理：** 支持批量交易導入、解析、聚合提交
- **實時狀態同步：** 實時跟蹤鏈上執行狀態，同步反饋發起方

### 4.4 合規清算引擎

**主要功能：**
- **實時合規校驗：** 交易發起後自動對交易對手方、交易金額、交易用途進行合規篩查
- **制裁名單篩查：** 對接 OFAC、UNSC 等全球制裁名單數據庫，支持實時更新
- **異常交易識別：** 基於機器學習模型，分析交易頻率、資金流向等多維度數據，識別可疑交易
- **清算結果存證：** 鏈上哈希存證 + 鏈下加密存儲，確保清算數據可追溯
- **近實時清算：** 跨境清算總延遲 ≤ 3.5 秒，清算台賬支持多格式導出（CSV/Excel/PDF）

### 4.5 跨鏈互通與資產流轉

**主要功能：**
- **跨鏈資產操作：** 支持同構/異構鏈間資產鎖定、解鎖、鑄造映射
- **原子性保障：** 確保跨鏈操作全部成功或全部回滾，防止資產丟失
- **多鏈余額查詢：** 統一查詢接口，屏蔽多鏈差異，實時查詢各鏈資產余額
- **跨鏈橋管理：** 插件化接入合規跨鏈橋，支持健康監控、異常告警、故障回滾

### 4.6 數據安全同步管理

**主要功能：**
- **傳輸加密：** TLS 1.3 / 國密 SM4 / 國際 AES-256 三重加密保障
- **存儲加密：** 敏感數據脫敏 + 加密存儲，HSM 硬件存儲私鑰
- **密鑰管理：** 密鑰分級管理，輪換周期 ≤ 90 天，支持密鑰生成、更新、銷毀全生命周期
- **鏈上存證：** 核心交易數據鏈上哈希存證，防篡改，可追溯
- **數據訪問審計：** 所有數據操作日誌留存 ≥ 7 年，滿足監管要求

### 4.7 用戶、角色與權限管理

**系統預設角色：**

| 角色 | 職責 | 權限範圍 |
|------|------|----------|
| 超級管理員 | 系統整體管理 | 最高操作權限 |
| 業務操作員 | 日常業務操作 | 交易發起、數據查詢、報表導出 |
| 合規管理員 | 合規規則配置與審核 | 合規管控相關操作 |
| 技術運維員 | 系統部署與維護 | 運維相關操作 |
| 接口對接員 | 外部接口對接調試 | 接口配置相關操作 |

**主要功能：**
- **用戶管理：** 支持銀行/機構/運營/管理員多類型用戶增刪改凍，實名認證與 KYC 綁定
- **身份映射：** 建立「銀行賬戶→法定身份→DID→錢包地址」四級映射
- **細粒度權限：** 功能/接口/數據/操作四維度權限配置
- **敏感操作多簽：** 路由策略修改、合規規則更新等敏感操作需多簽審批

### 4.8 系統監控與運維管理

**主要功能：**
- **監控指標：** 實時監控 TPS、響應時間、成功率、鏈節點狀態
- **多渠道告警：** 支持郵件、短信、企業微信等多渠道告警通知
- **可視化展示：** Prometheus + Grafana 實現監控數據可視化
- **配置管理：** 通過 Apollo 配置中心動態管理插件、路由策略、合規規則，支持灰度發布
- **報表管理：** 多維度數據統計，支持運營/交易/合規/清算報表生成與加密導出

## 5. 安裝與部署

### 5.1 部署架構

本軟件採用 **Docker + Kubernetes** 容器化部署，支持高可用集群模式。

```
┌──────────────────────────────────────────────────┐
│                  Kubernetes 集群                  │
│                                                  │
│  ┌──────────────┐    ┌──────────────┐            │
│  │  網關節點 1   │    │  網關節點 2   │  ...       │
│  └──────────────┘    └──────────────┘            │
│                                                  │
│  ┌──────────────────────────────────────────┐    │
│  │            數據層（StatefulSet）           │    │
│  │  MySQL 主從 │ Redis 集群 │ Elasticsearch  │    │
│  └──────────────────────────────────────────┘    │
└──────────────────────────────────────────────────┘
            │
      負載均衡（Nginx / HAProxy）
            │
        外部訪問入口
```

### 5.2 部署步驟

**步驟一：環境準備**

1. 確認服務器滿足第 2 章硬件與軟件要求
2. 安裝 Docker 及 Kubernetes 環境
3. 配置 HSM 硬件安全模塊並完成初始化
4. 準備數據庫（MySQL/PostgreSQL）、Redis、RocketMQ 等中間件

**步驟二：配置文件準備**

1. 修改 `application.yml` 配置數據庫連接、Redis 地址、RocketMQ 地址
2. 配置多鏈節點地址（`chain-nodes.yml`）
3. 配置合規規則（`compliance-rules.yml`），導入制裁名單數據
4. 配置 HSM 密鑰參數（`hsm.yml`）

**步驟三：數據庫初始化**

```bash
# 執行數據庫初始化腳本
mysql -u root -p < init/schema.sql
mysql -u root -p < init/data.sql
```

**步驟四：容器化部署**

```bash
# 拉取鏡像
docker pull chainfusion/gateway:v1.0

# 使用 Kubernetes 部署
kubectl apply -f deploy/k8s/

# 確認 Pod 狀態
kubectl get pods -n chainfusion
```

**步驟五：健康檢查**

```bash
# 檢查服務健康狀態
curl https://<gateway-host>/actuator/health

# 預期返回
{"status":"UP"}
```

### 5.3 升級說明

1. 備份當前配置文件與數據庫
2. 通過 Apollo 配置中心下發新配置（支持灰度發布）
3. 滾動更新 Kubernetes Deployment，確保零停機升級
4. 升級完成後執行回歸測試，確認核心功能正常

## 6. 使用說明

### 6.1 系統登錄

1. 通過瀏覽器訪問管理控制台：`https://<console-host>/login`
2. 使用管理員賬號密碼登錄，首次登錄需修改初始密碼
3. 系統支持雙因素認證（2FA），建議開啟以提升安全性

### 6.2 主要操作流程

#### 6.2.1 發起跨境交易

1. 進入「交易管理」模塊
2. 填寫交易基本信息（交易對手方、金額、目標鏈、資產類型）
3. 系統自動進行合規前置校驗（AML/CFT、制裁名單篩查）
4. 校驗通過後，系統自動選擇最優路由並轉發交易
5. 實時查看交易狀態，等待鏈上確認與清算結果回調

#### 6.2.2 配置路由規則

1. 進入「系統配置 → 路由管理」
2. 新增或修改路由規則（按交易類型、資產類型、目標鏈等維度配置）
3. 路由策略修改需通過多簽審批流程
4. 審批通過後規則即時生效，無需重啟服務

#### 6.2.3 合規規則管理

1. 進入「合規管理 → 規則配置」
2. 配置 AML/CFT、旅行規則、多司法轄區合規規則
3. 導入或更新制裁名單（支持 OFAC、UNSC 等數據源）
4. 生成並導出合規報表（支持多語言、多轄區）

#### 6.2.4 系統監控

1. 進入「監控中心」查看 Grafana 可視化儀表盤
2. 監控指標包括：TPS、響應時間、交易成功率、鏈節點健康度
3. 配置告警閾值和告警通知渠道（郵件/短信/企業微信）

### 6.3 注意事項

## 8. 安全說明

### 8.1 加密體系

| 層次 | 算法 | 說明 |
|------|------|------|
| 傳輸加密 | TLS 1.3 | 所有外部通信強制啟用 TLS 1.3 |
| 存儲加密 | AES-256 / 國密 SM4 | 敏感數據加密存儲，支持國密與國際算法切換 |
| 簽名算法 | ECDSA secp256k1 / 國密 SM2 | 交易簽名驗證，防止未授權操作 |
| 密鑰存儲 | HSM 硬件安全模塊 | 私鑰由 HSM 存儲，滿足 FIPS 140-2 Level 3 |

### 8.2 防攻擊措施

- **防重放攻擊：** 時間戳 + nonce 機制，請求有效期 5 分鐘
- **防 DDoS：** Redis + Lua 腳本實現接口限流，集成 WAF 過濾惡意流量
- **防注入攻擊：** 參數化查詢，拒絕 SQL 注入、XSS 等非法請求
- **入侵檢測：** 集成 IDS 系統，實時監測異常訪問行為並告警

### 8.3 合規安全

- 滿足《網絡安全等級保護基本要求》三級及以上
- 適配 GDPR、中國《數據出境安全評估辦法》、MiCA 法案、AML/CFT 要求
- 所有操作日誌、交易數據留存 ≥ 7 年
- 數據訪問遵循最小權限原則，越權操作實時告警

## 10. 運維管理

### 10.1 監控體系

- **指標監控：** Prometheus 採集，Grafana 可視化展示 TPS、響應時間、成功率、節點健康度
- **日誌管理：** ELK（Elasticsearch + Logstash + Kibana）集中日誌采集與分析
- **告警通知：** AlertManager 配置多渠道告警（郵件、短信、企業微信），告警響應時間 ≤ 1 分鐘

### 10.2 日常維護

- **健康巡檢：** 每日自動執行鏈節點連通性、數據庫、緩存健康巡檢
- **數據備份：** 數據庫每日全量備份，交易日誌實時增量備份；備份數據加密存儲
- **密鑰輪換：** 每 90 天執行一次密鑰輪換，確保密鑰安全
- **合規名單更新：** 制裁名單（OFAC、UNSC）每日自動更新

### 10.3 故障處理

**節點故障：**
1. 系統自動檢測主節點故障，觸發容災切換至備用節點
2. 自動切換完成後通過告警渠道通知運維人員
3. 運維人員排查主節點故障原因，修復後可手動或自動恢復

**交易失敗：**
1. 失敗交易自動進入 RocketMQ 重試隊列
2. 按指數退避策略自動重試（最多 3 次）
3. 重試均失敗後觸發告警，人工介入處理

**系統異常：**
1. Kubernetes 自動重啟異常 Pod
2. 若整個 Deployment 不可用，觸發高優先級告警
3. 運維人員按故障排查手冊進行處理

## 11. 常見問題

**Q1：新增一條區塊鏈的接入需要多長時間？**

A：通過插件化擴展機制，新增鏈適配周期通常 ≤ 1-2 週。只需開發對應的適配插件，無需修改核心模塊代碼。

**Q2：交易失敗後如何處理？**

A：失敗交易會自動進入重試隊列，按指數退避策略自動重試最多 3 次。若重試均失敗，系統會觸發告警並通知運維人員，同時在管理控制台中標記該筆異常交易，支持手動重試或回滾。

**Q3：如何保證跨鏈交易的原子性？**

A：系統通過跨鏈原子性保障機制確保跨鏈操作全部成功或全部回滾。若跨鏈過程中發生異常，系統會自動觸發回滾流程，防止資產丟失或狀態不一致。

**Q4：合規規則更新後是否需要重啟系統？**

A：不需要。合規規則通過 Apollo 配置中心動態配置，更新後即時生效，無需重啟服務。

**Q5：系統是否支持國密算法？**

A：支持。系統同時支持國密（SM2/SM4）與國際通用算法（ECDSA/AES-256），可根據業務需求動態切換，滿足國內合規要求。

**Q6：歷史交易數據能保存多長時間？**

A：系統要求所有交易數據、合規數據、操作日誌留存 ≥ 7 年，滿足金融監管要求。歷史數據採用加密存儲，並支持查詢與審計導出。
