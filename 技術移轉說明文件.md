# 技術移轉說明文件

> **文件版本：** v1.0
> **建立日期：** 2026-04-28
> **適用範圍：** Cold Wallet（Aegis Vault + Keyvault）、Hot Wallet（HSM-based custody）

---

## 一、文件目的

本文件旨在說明冷熱錢包系統於技術移轉情境下，哪些技術資產**可隨程式碼一同移轉**，以及哪些屬於**不可移轉項目**（需於目標環境重新建立、申請或生成）。適用對象包含：接收方技術團隊、資安稽核人員、專案管理人員。

---

## 二、系統組成速覽

```
Cold Wallet
├── 應用層：aegis-vault-api / aegis-vault-app / aegis-vault-admin / aegisvault-transaction
├── Keyvault 核心：keyvault-api-service / keyvault-ethereum-service / keyvault-kms-service
├── 行動授權：kv-auth-app（ReactJS + Expo）
└── 斷網簽署：hk-hsm-ops（Next.js，air-gapped）

Hot Wallet
└── hotwallet（Golang，支援 SoftHSM / CloudHSM / Thales）

共用基礎設施
├── keyvault-gateway → walletos（多鏈 JSON-RPC）
├── crypto-event-subscriber / crypto-event-ws / transaction-listener（事件鏈路）
└── stablecoin-studio / code-push-server（附屬子系統）
```

## 四、不可移轉範圍

以下項目**無法直接隨程式碼移轉**，接收方必須於目標環境重新建立、申請或生成。強行複製這些項目將導致嚴重安全風險或功能失效。

### 4.1 HSM 硬體裝置與金鑰材料（最高風險）

| 項目 | 說明 | 處置方式 |
|---|---|---|
| HSM 硬體裝置（Thales / CloudHSM / SoftHSM token） | 物理設備或雲端 HSM 實例，金鑰存於硬體安全模組內，無法匯出 | 目標環境須另行採購或申請新 HSM 實例 |
| HSM Partition PIN / SO PIN | 存取 HSM 的管理憑證 | 需於新 HSM 上重新初始化並設定 |
| PKCS#11 金鑰物件（CKO_PRIVATE_KEY） | 冷熱錢包的簽署私鑰，受 HSM 保護，不可匯出 | 需在目標 HSM 上重新生成；原有地址與資產必須先完成遷移 |
| SoftHSM token 資料夾 | 本地測試用 token，含私鑰材料 | 移轉後需重新初始化，不可直接複製 |

> **重要：** 私鑰材料的複製或移動等同於密鑰外洩，嚴禁在任何情境下直接搬移。

### 4.2 憑證與 TLS 材料

| 項目 | 說明 | 處置方式 |
|---|---|---|
| TLS/SSL 憑證（含私鑰） | 綁定特定網域/IP，具有效期 | 目標環境須重新申請或以 Let's Encrypt / 自簽 CA 簽發 |
| mTLS 用戶端憑證 | 服務間互信所用，綁定原環境 PKI | 需以目標環境 CA 重新簽發 |
| HSM 裝置憑證 | 由 HSM 廠商簽發，綁定特定硬體 | 無法移轉，隨硬體取得 |

### 4.3 第三方服務憑證與 API 金鑰

| 服務 | 項目 | 處置方式 |
|---|---|---|
| AWS（CloudHSM / S3 / SES） | IAM Access Key、Secret、CloudHSM cluster ID | 目標帳號重新建立 IAM 角色並申請 |
| Google Drive / GCP | OAuth2 Client ID/Secret、Service Account JSON | 目標 GCP 專案重新建立 |
| Mailgun / SendGrid | API Key、Domain 驗證 | 重新申請並完成 DNS 驗證 |
| reCAPTCHA | Site Key / Secret Key | 新網域重新申請 |
| CryptoAPIs | API Key（鏈上事件訂閱） | 重新申請或移轉帳號 |
| Kafka（MSK / 自建） | Bootstrap servers、憑證、帳密 | 目標環境重建 Kafka Cluster 並重新設定 |
| Push Notification（FCM / APNs） | Firebase Server Key、APNs 憑證 | 重新申請，APNs 需對應 Apple 開發者帳號 |

### 4.4 環境機密設定檔

| 項目 | 說明 | 處置方式 |
|---|---|---|
| `.env`（所有 NestJS 服務） | 含 DB 密碼、JWT Secret、API Keys 等 | 依各服務 README 重新填寫，所有 Secret 須換新 |
| `config/config.yml`（Golang 服務） | 含 DB 連線字串、Kafka、HSM 設定 | 從 `config.yml.bak` 重新建立並填入目標環境值 |
| JWT / Session Secret | 用於 Token 簽發，具環境唯一性 | 強制重新生成（使用足夠長度的隨機值） |
| DB 加密金鑰（若有欄位層加密） | 保護資料庫中敏感欄位 | 重新生成；若需遷移資料，須先解密再重新加密 |

### 4.5 資料庫資料

| 項目 | 說明 | 處置方式 |
|---|---|---|
| 使用者帳號與密碼雜湊 | 含業務資料，屬資料主權範疇 | 需依資料移轉協議辦理，不屬技術移轉範疇 |
| 錢包地址與交易歷史 | 鏈上可重建部分，鏈下記錄需協議 | 另行議定資料移轉計畫 |
| KMS/Keyvault 元數據 | 記錄金鑰派生路徑與錢包對應關係 | 配合 HSM 金鑰遷移計畫同步處理 |

### 4.6 行動 App 簽署憑證

| 項目 | 說明 | 處置方式 |
|---|---|---|
| Android Keystore（`kv-auth-app`） | App 簽署金鑰，綁定 Google Play Developer 帳號 | 目標方需以自有 Keystore 重新簽署並上架 |
| Apple Distribution Certificate + Provisioning Profile | 綁定 Apple Developer 帳號 | 目標方需使用自有 Apple 帳號申請並重新簽署 |

### 4.7 網域、DNS 與 IP 綁定設定

- 原有網域名稱及 DNS 設定（不隨程式碼移轉）
- reCAPTCHA、OAuth callback URL 中的原網域
- TLS 憑證中的 SAN（Subject Alternative Name）
- 上述項目均需在目標環境配合新網域重新設定

## 五、灰色地帶：需視情況決定

以下項目依移轉合約與雙方協議而定：

| 項目 | 說明 |
|---|---|
| Kafka Topics 命名慣例與 Schema Registry | 技術上可移轉，但若與現有系統共用 Kafka 需特別協商 |
| Docker Registry / Image | 若接收方可存取原 Registry，可直接拉取；否則需重新 build |
| `hk-deployment` 中的外部 IP / hostname 設定 | 需依目標網路環境調整，不可直接套用 |
| Monitoring / Alerting 設定（Grafana / Prometheus） | 設定可移轉，但告警收件人、Webhook URL 需重設 |
| Code Push Server 既有 Bundle | App bundle 可移轉，但 App 更新金鑰需重新建立 |

## 六、移轉前置檢查清單

接收方在正式接手前，應確認以下項目已就緒：

### 基礎設施
- [ ] 目標 HSM 已採購/申請並完成初始化
- [ ] 目標 Kafka Cluster 已建立
- [ ] 目標 PostgreSQL 資料庫已建立
- [ ] 網域與 DNS 已設定，TLS 憑證已申請

### 第三方服務
- [ ] AWS / GCP / Azure 帳號已建立，對應服務已開通
- [ ] Mailgun / reCAPTCHA / CryptoAPIs 帳號已申請
- [ ] Push Notification（FCM/APNs）憑證已申請

### 行動 App
- [ ] Apple Developer 帳號已備妥
- [ ] Google Play Console 帳號已備妥
- [ ] App 簽署 Keystore 已生成

### 機密管理
- [ ] 所有 `.env` 與 `config.yml` 已依目標環境重新填寫（不得沿用原環境值）
- [ ] JWT Secret、加密金鑰等已重新生成
- [ ] HSM Partition PIN 已在新設備上設定

## 七、安全注意事項

1. **私鑰不可移動**：任何情況下均不得直接複製 HSM 中的私鑰材料。資產遷移需透過區塊鏈交易完成，而非金鑰複製。
2. **機密輪換**：移轉後，所有共享過的密碼、API Key、JWT Secret 必須立即輪換，即使移轉協議中包含這些資訊。
3. **最小權限原則**：目標環境的 IAM Role、DB 帳號、Kafka ACL 需重新以最小權限原則設定，不得沿用原環境的寬鬆設定。
4. **稽核日誌**：確保目標環境具備等同或更強的稽核日誌能力（HSM audit log、API access log、DB audit）。
5. **斷網環境（hk-hsm-ops）**：若接收方需部署斷網簽署模式，air-gapped 主機的隔離措施需重新規劃，不可假設原環境的網路隔離設計可直接沿用。
