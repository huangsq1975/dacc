## 1. 项目概述

### 1.1 背景

当前金砖国家间跨境贸易、投融资合作持续深化，数字资产与 CBDC（如 eCNY）应用场景不断拓展。传统网关存在以下痛点：

- 多链协议不兼容，无法统一接入公链与联盟链
- 跨境清算效率低，延迟高、成本高
- 数据安全同步不足，存在数据一致性风险
- 合规管控薄弱，难以满足多司法辖区监管要求

分布式多链融合网关（Chain Fusion）作为链上链下融合的核心枢纽，打通不同区块链与传统金融系统（CIPS、SWIFT、银行核心系统）的协同壁垒，实现数字资产交易、清算全流程的高效、安全、合规流转。

### 1.2 核心建设目标

| 目标 | 说明 |
|------|------|
| 多链协议统一适配 | 支持 Conflux、ETH、BSC 等主流公链与 CIPS 联盟链的无缝接入，打破"链岛"壁垒 |
| 跨境清算效率优化 | 跨境交易近实时处理，延迟 ≤ 3.5 秒，清算成本降低 60% 以上 |
| 金融级合规管控 | 内嵌 AML/CFT、制裁名单筛查，满足多司法辖区监管要求 |
| 数字资产全生命周期管理 | 支持 RWA 发行、数字资产托管、交易流转、清算结算全流程 |
| 高可用与安全 | 7×24 小时稳定运行，年可用性 ≥ 99.99%，满足金融级安全标准 |

### 1.3 适用范围

- **接入链类型：** Conflux、ETH、BSC 等主流公链，CIPS 相关联盟链，合规 DeFi 平台
- **业务场景：** 跨境支付清算、RWA 代币化发行与托管、贸易融资单据存证、跨链资产映射
- **对接系统：** COBOL 大型机、分布式核心系统、Open Banking 平台

### 1.4 术语定义

| 术语 | 定义 |
|------|------|
| 多链融合网关 | 连接不同区块链与传统金融系统，实现协议适配、数据同步、交易转发、合规管控的核心基础设施 |
| CIPS | 人民币跨境支付系统，提供跨境人民币清算、结算服务 |
| mBridge | 多边央行数字货币桥项目，实现不同 CBDC 的跨境协同清算 |
| eCNY | 数字人民币，中国人民银行发行的法定数字货币 |
| RWA | 现实资产代币化，将大宗商品、基建项目收益权等现实资产转化为可流转代币 |
| DID | 分布式身份标识，实现链上链下身份统一映射 |
| HSM | 硬件安全模块，用于私钥的安全存储与管理 |

## 4. 核心模块详细设计

### 4.1 协议适配子模块

#### 4.1.1 架构设计

采用"插件化 + 抽象封装"三层架构：

```
外部请求（银行系统 / 区块链节点）
        │
┌───────▼─────────┐
│   协议抽象层     │  定义统一协议接口，屏蔽异构差异
└───────┬─────────┘
        │
┌───────▼─────────┐
│   插件适配层     │  每个区块链 / 金融协议对应一个适配插件
│                  │  Conflux（JSON-RPC）/ ETH（Web3.js）/
│                  │  BSC（Web3.js）/ CIPS（自定义 SDK）
└───────┬─────────┘
        │
┌───────▼─────────┐
│   协议转换层     │  格式转换：JSON/ABI ↔ COBOL/XML/Protobuf
└─────────────────┘
```

#### 4.1.2 核心功能

- **多链协议封装：** 对主流公链、联盟链的原生协议进行统一抽象封装，提供标准化接口
- **协议转换：** 支持链上链下数据格式双向转换，支持 EBCDIC/UTF-8 编码转换
- **插件化扩展：** 通过 SPI（Service Provider Interface）机制加载适配插件，支持热插拔，新增链适配周期 ≤ 1-2 周
- **协议校验：** 格式校验、完整性校验，拒绝非法协议请求

#### 4.1.3 统一内部消息格式（Protobuf）

```protobuf
message GatewayMessage {
    string request_id     = 1;  // 请求唯一 ID
    int64  timestamp      = 2;  // UTC 毫秒时间戳
    string source_chain   = 3;  // 源链标识
    string target_chain   = 4;  // 目标链标识
    string asset_type     = 5;  // 资产类型
    string payload        = 6;  // 业务数据（JSON 序列化）
    string signature      = 7;  // 请求签名
    string bank_account   = 8;  // 银行账户（可选）
}
```

```
请求接入
    │
    ▼
请求解析（提取源链、目标链、资产类型等关键信息）
    │
    ▼
规则匹配（基于预设规则：资产类型 / 网络性能 / 合规要求）
    │
    ▼
负载均衡（在多个可用目标链节点间分配流量）
    │
    ▼
目标链选择（综合匹配结果与负载状态，智能决策最优节点）
    │
    ▼
交易转发
```

#### 4.2.2 健康度评分算法

```
健康度 = (连通率 × 40% + 确认延迟达标率 × 30% + 跨链桥成功率 × 20% + 合规状态 × 10%) × 100
```

- **数据更新频率：** 普通链 10 秒/次，高优先级链（CIPS 联盟链）1 秒/次
- **路由优先策略：** 合规优先 → 效率优先（核心交易）/ 成本优先（非核心交易）→ 健康度优先

#### 4.2.3 设计要点

- **动态配置：** 路由规则支持动态下发与热更新，无需重启服务
- **算法支持：** 内置轮询、加权轮询、最少连接等多种负载均衡算法
- **容灾切换：** 主节点故障时自动切换至备用节点，恢复时间 ≤ 5 分钟

### 4.3 合规控制子模块

#### 4.3.1 身份映射体系

建立"银行账户 → 法定身份 → DID → 钱包地址"四级映射：

```
银行客户完成 KYC
    │
    ▼
生成 / 绑定合规钱包地址
    │
    ▼
映射关系链上哈希存证（不可篡改）
    +
映射关系链下 AES-256 加密存储（仅授权节点可查询）
```

#### 4.3.2 旅行规则适配

- 交易发起时，自动提取双方钱包地址对应的法定身份信息，同步至监管系统
- 拒绝未绑定身份的匿名交易，仅允许合规身份关联的钱包地址参与

#### 4.3.3 多辖区合规规则库

| 辖区 | 适配规则 |
|------|----------|
| 欧盟 | GDPR、MiCA 法案、MiFID II |
| 中国 | 《数据出境安全评估办法》、AML、eCNY 相关规范 |
| 香港 | SFC 报表要求、《个人数据（隐私）条例》、HKMA 监管 |
| 全球 | FATF 旅行规则、OFAC / UNSC 制裁名单、CFT |
| 金砖国家 | mBridge 合规要求、NDB 相关金融监管 |

#### 4.3.4 合规报表生成

- 支持自动生成多语言、多辖区合规报表
- 报表生成时效 ≤ 1 小时
- 包含交易合规汇总、异常交易明细、制裁名单筛查结果

### 4.4 安全防护子模块

#### 4.4.1 传输安全

- 网关与银行系统、多链节点通信采用 TLS 1.3 加密
- 敏感字段（身份信息、交易金额）额外采用国密 SM4 加密

#### 4.4.2 存储安全

- 私钥采用 HSM 硬件安全模块存储，禁止明文存储
- 交易日志、合规数据加密存储，留存期 ≥ 7 年
- 密钥轮换周期 ≤ 90 天

#### 4.4.3 访问控制

- 基于 RBAC 模型实现权限分级，遵循最小权限原则
- 敏感操作（路由策略修改、合规规则更新）需多签审批
- OAuth 2.0 + JWT 实现无状态认证与授权管理

#### 4.4.4 防攻击设计

- 时间戳 + nonce 防止重放攻击，请求有效期 ≤ 30 秒
- 限流阈值动态调整（基于业务峰值），Redis + Lua 实现高性能限流
- WAF 拦截 SQL 注入、XSS、DDoS 等恶意请求
- 部署 IDS 入侵检测系统

### 4.5 交易处理子模块

#### 4.5.1 核心流程（跨境交易清算）

```
① 银行内部系统发起跨境交易指令
② 接入层：请求校验 + 负载均衡 + 协议格式转换
③ 协议适配子模块：协议格式转换 + 合法性校验
④ 交易处理子模块：参数校验 + 权限校验 + 余额校验
⑤ 合规校验子模块：制裁名单筛查 + 异常交易识别
⑥ 动态路由子模块：选择最优路由路径（CIPS / mBridge / 公链）
⑦ 系统对接子模块：转发交易指令至对应系统
⑧ 接收交易执行结果 → 更新交易状态 → 写入数据层
⑨ 将交易结果通过接入层反馈至银行内部系统
```

#### 4.5.2 异常处理

- 支持交易重试（最大重试次数可配置），基于 RocketMQ 重试队列
- 支持手动干预与回滚操作
- 记录完整异常日志，包含异常原因、堆栈、上下文信息

## 6. 数据模型设计

### 6.1 核心数据实体

#### Chain（区块链节点）

```sql
CREATE TABLE chain (
    id            BIGINT       PRIMARY KEY AUTO_INCREMENT,
    name          VARCHAR(64)  NOT NULL COMMENT '链名称',
    type          VARCHAR(32)  NOT NULL COMMENT '链类型：PUBLIC/ALLIANCE',
    protocol      VARCHAR(64)  NOT NULL COMMENT '协议类型：JSON-RPC/gRPC 等',
    rpc_url       VARCHAR(256) NOT NULL COMMENT '节点接入地址',
    health_score  DECIMAL(5,2) DEFAULT 100 COMMENT '健康度评分',
    status        TINYINT      NOT NULL COMMENT '状态：1-启用 2-维护 3-禁用',
    create_time   DATETIME     NOT NULL,
    update_time   DATETIME     NOT NULL
);
```

#### Asset（数字资产）

```sql
CREATE TABLE asset (
    id               BIGINT       PRIMARY KEY AUTO_INCREMENT,
    name             VARCHAR(64)  NOT NULL COMMENT '资产名称',
    symbol           VARCHAR(32)  NOT NULL COMMENT '资产符号',
    chain_id         BIGINT       NOT NULL COMMENT '所属链 ID（FK -> chain）',
    contract_address VARCHAR(256) COMMENT '合约地址',
    decimal_places   INT          NOT NULL COMMENT '精度',
    asset_type       VARCHAR(32)  NOT NULL COMMENT '资产类型：TOKEN/RWA/CBDC',
    status           TINYINT      NOT NULL,
    create_time      DATETIME     NOT NULL
);
```

#### Transaction（跨链交易）

```sql
CREATE TABLE transaction (
    id               BIGINT        PRIMARY KEY AUTO_INCREMENT,
    gateway_tx_id    VARCHAR(64)   UNIQUE NOT NULL COMMENT '网关交易 ID',
    source_chain_id  BIGINT        NOT NULL COMMENT '源链 ID（FK -> chain）',
    target_chain_id  BIGINT        NOT NULL COMMENT '目标链 ID（FK -> chain）',
    asset_id         BIGINT        NOT NULL COMMENT '资产 ID（FK -> asset）',
    from_address     VARCHAR(256)  NOT NULL COMMENT '源地址',
    to_address       VARCHAR(256)  NOT NULL COMMENT '目标地址',
    amount           DECIMAL(36,18) NOT NULL COMMENT '交易金额',
    source_tx_hash   VARCHAR(256)  COMMENT '源链交易哈希',
    target_tx_hash   VARCHAR(256)  COMMENT '目标链交易哈希',
    status           VARCHAR(32)   NOT NULL COMMENT '状态：PENDING/PROCESSING/COMPLETED/FAILED',
    compliance_result VARCHAR(32)  NOT NULL COMMENT '合规校验结果：PASS/FAIL/REVIEW',
    create_time      DATETIME      NOT NULL,
    update_time      DATETIME      NOT NULL
);
```

#### User（用户）

```sql
CREATE TABLE user (
    id            BIGINT       PRIMARY KEY AUTO_INCREMENT,
    username      VARCHAR(64)  UNIQUE NOT NULL,
    password_hash VARCHAR(256) NOT NULL,
    api_key       VARCHAR(128) UNIQUE,
    bank_account  VARCHAR(64)  COMMENT '绑定银行账户',
    wallet_address VARCHAR(256) COMMENT '绑定钱包地址',
    did           VARCHAR(256) COMMENT 'DID 分布式身份标识',
    kyc_status    TINYINT      NOT NULL COMMENT 'KYC 状态',
    status        TINYINT      NOT NULL,
    create_time   DATETIME     NOT NULL
);
```

#### Role / Permission（角色权限）

```sql
CREATE TABLE role (
    id          BIGINT      PRIMARY KEY AUTO_INCREMENT,
    name        VARCHAR(64) UNIQUE NOT NULL COMMENT '角色名',
    description VARCHAR(256),
    create_time DATETIME    NOT NULL
);

CREATE TABLE permission (
    id          BIGINT      PRIMARY KEY AUTO_INCREMENT,
    code        VARCHAR(128) UNIQUE NOT NULL COMMENT '权限编码',
    name        VARCHAR(64)  NOT NULL COMMENT '权限名称',
    type        VARCHAR(32)  NOT NULL COMMENT '类型：FUNC/API/DATA/OP'
);
```

### 6.2 核心关系映射

- `Chain 1:N Asset` — 一条链包含多种资产
- `User N:M Role` — 用户支持多角色（单用户多角色）
- `Role N:M Permission` — 角色包含多种权限
- `Transaction N:1 Chain（source/target）` — 交易关联源链和目标链
- `Transaction N:1 Asset` — 交易关联具体资产

### 6.3 数据存储策略

| 数据类型 | 存储方案 | 说明 |
|----------|----------|------|
| 交易数据 | MySQL 主从架构 | 主库写入，从库读取；热点数据缓存于 Redis |
| 合规数据 | MySQL + Redis | 制裁名单存 Redis 快速查询，规则和报表存 MySQL |
| 配置数据 | MySQL + Redis | 写入 MySQL，缓存 Redis 快速加载 |
| 日志数据 | Elasticsearch | 支持全文检索，留存 ≥ 7 年 |
| 监控数据 | InfluxDB | 时序数据库，Grafana 可视化 |
| 私钥 | HSM 硬件设备 | 严禁明文存储 |

## 7. 技术选型

### 7.1 网关核心框架（推荐方案）

| 方案 | 核心技术栈 | 性能 | 优势 | 适用场景 |
|------|-----------|------|------|----------|
| **Apache APISIX（首选）** | OpenResty（Nginx + LuaJIT）+ etcd | 极高，单实例 QPS 可达数万至十万级 | 基于 Nginx 极致性能，毫秒级动态配置，插件生态丰富，云原生友好 | 超大规模流量、高并发、动态配置场景 |
| Spring Cloud Gateway（备选） | Java、Spring WebFlux、Netty | 中高，单实例 QPS 约 1.5-2 万 | 与 Spring 生态无缝集成，Java 团队友好，复杂业务逻辑定制方便 | Java 技术栈主导、业务逻辑复杂场景 |
| 混合架构（超大规模） | APISIX（流量网关）+ Spring Cloud Gateway（业务网关） | 极高 | 前端 APISIX 负责流量、安全、协议转换；后端 Spring Cloud Gateway 负责复杂业务逻辑 | 超大规模多链融合平台 |

**最终建议：** 优先选择 **Apache APISIX** 作为网关核心框架；若团队为 Java 技术栈，选择 **Spring Cloud Gateway + Netty**；超大规模场景采用混合架构。

## 9. 部署架构

### 9.1 环境体系

| 环境 | 说明 |
|------|------|
| 开发环境（DEV） | 开发人员本地联调 |
| 测试环境（TEST） | 功能测试、集成测试 |
| 预生产环境（UAT） | 性能测试、用户验收 |
| 生产环境（PROD） | 正式业务运行 |

### 9.2 容器化部署

- 基于 **Docker** 容器化封装，**Kubernetes** 进行编排管理
- 支持水平弹性扩展，通过增加 Pod 副本数提升系统性能
- 多节点集群部署，确保高可用

### 9.3 CI/CD 流水线

```
代码提交（Git Push）
    │
    ▼
触发 CI 流水线（Jenkins / GitLab CI）
    │
    ▼
自动化构建（Maven / Gradle）
    │
    ▼
单元测试 + 代码扫描（SonarQube）
    │
    ▼
构建 Docker 镜像 → 推送镜像仓库
    │
    ▼
自动部署至测试环境
    │
    ▼
集成测试 → 通过后 → 部署至预生产 → 人工确认 → 部署至生产
```

## 10. 测试计划

### 10.1 测试策略

| 测试类型 | 目标 | 说明 |
|----------|------|------|
| 单元测试 | 验证单个模块的功能正确性 | 核心模块覆盖率 ≥ 80% |
| 集成测试 | 验证模块间的交互与接口调用 | 消除集成风险，验证跨模块数据流转 |
| 系统测试 | 验证整个网关系统是否满足业务需求 | 覆盖所有核心业务场景 |
| 性能测试 | 验证高并发场景下的 TPS、响应时间 | 模拟 1000/5000 TPS 压测 |
| 安全测试 | 渗透测试、漏洞扫描 | SQL 注入、XSS、重放攻击、DDoS 等 |
| 合规测试 | 验证合规校验规则的准确性 | 制裁名单命中率、误报率 |

### 10.2 测试用例设计原则

- **全场景覆盖：** 针对每个核心功能模块，设计正向、反向、边界值测试用例
- **异常与恢复专项：** 重点设计跨链交易异常处理和恢复测试，验证极端情况下的数据一致性
- **多链兼容性测试：** 每条新接入链上线前必须通过完整兼容性测试套件

### 10.3 验收标准

| 指标 | 验收标准 |
|------|----------|
| 功能完整性 | 所有功能需求 100% 实现，无 P0/P1 级缺陷 |
| 性能 | 并发 ≥ 1000 TPS，响应时间 ≤ 500ms |
| 可用性 | 压测期间系统可用性 ≥ 99.9% |
| 安全 | 无高危安全漏洞，渗透测试通过 |
| 合规 | 制裁名单命中准确率 ≥ 99.99% |

## 12. 风险评估

| 风险类型 | 具体风险 | 应对措施 |
|----------|----------|----------|
| 技术风险 | 多链协议适配不兼容，导致交易失败 | 前期充分的多链兼容性测试，插件灰度发布机制，发现问题快速回滚 |
| 技术风险 | 高并发下系统性能不足 | 采用 Apache APISIX 高性能框架，热点数据缓存，水平扩展 |
| 合规风险 | 多辖区监管规则更新，合规能力未及时迭代 | 建立监管政策跟踪机制，合规规则库动态更新，无需重启服务 |
| 运营风险 | 链节点故障、跨链桥中断导致业务中断 | 多节点集群 + 备用跨链桥，动态路由自动切换，RocketMQ 重试队列 |
| 安全风险 | 网关被攻击、数据泄露 | WAF + IDS 部署，敏感数据加密存储，HSM 密钥管理，定期渗透测试 |
| 集成风险 | 与银行现有核心系统（COBOL 大型机）对接失败 | 开发专用适配中间件，采用"接口仿真测试→小流量联调→全量上线"渐进式集成策略 |
| 系统稳定性风险 | 系统故障、节点宕机 | 分布式架构节点冗余，容灾自动切换，定期备份与恢复测试 |
